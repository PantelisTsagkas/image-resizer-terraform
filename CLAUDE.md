## What this is

A serverless, event-driven image processing pipeline on AWS, provisioned entirely with Terraform. Browser uploads an image via presigned S3 URL â†’ S3 event triggers a Pillow-based resizer Lambda â†’ three variants (thumb/medium/large) land in an outputs bucket â†’ frontend polls an API Lambda for presigned GET URLs. It's a portfolio/demo project, not a hardened production service (see README's "Production Hardening" section for known gaps: open CORS, no auth, no WAF, etc. â€” don't silently "fix" these unless asked, they're intentional scope boundaries for a demo).

## Repository layout

```
terraform/   â†’ all AWS infra (Terraform, single source of truth)
lambda/      â†’ resizer Lambda source (handler.py) â€” this is the only Lambda with a real source file
frontend/    â†’ single-file HTML/JS UI, no build step, deployed to GitHub Pages
```

The **API Lambda** (presign + results endpoints) has no source file on disk â€” its Python code is inlined directly inside `terraform/api_gateway.tf` as a heredoc in an `archive_file` data source (see the `source { content = <<-EOT ... EOT }` block). To change API Lambda behavior, edit that heredoc, not a file in `lambda/`.

## Commands

All Terraform commands run from `terraform/`:

```bash
terraform init      # first-time setup / after adding providers
terraform plan       # preview changes
terraform apply      # provision everything (~60s)
terraform destroy    # tear down (buckets have force_destroy = true, so non-empty buckets delete cleanly)
```

There is no test suite, linter, or CI pipeline for the Terraform or Lambda code in this repo. The only GitHub Actions workflow (`.github/workflows/pages.yml`) deploys `frontend/` to GitHub Pages on push to `main`, injecting `API_BASE_URL` via `sed` from the `API_BASE_URL` repo/environment variable.

After `terraform apply`, copy the `api_base_url` output into the `API_BASE_URL` const near the top of the `<script>` in `frontend/index.html` (currently hardcoded to a live API Gateway URL â€” update it, don't leave stale values).

To run the frontend locally, it must be served over HTTP (not `file://`, which Chrome's CORS policy blocks):

```bash
cd frontend && python3 -m http.server 8080
```

Local Lambda dependency testing (Pillow is delivered via a Lambda layer at runtime, not bundled â€” `lambda/requirements.txt` exists only for local dev):

```bash
uv venv && uv pip install -r lambda/requirements.txt
```

## Architecture notes that span multiple files

- **Two separate Lambdas, two separate IAM roles**, each scoped to least privilege and defined in different files: the resizer's role is in `iam.tf` (read `uploads/*`, write `outputs/*`), the API Lambda's role is defined inline in `api_gateway.tf` (write `uploads/originals/*` only, read/list `outputs/resized/*` only). Neither can delete objects or touch the other's primary bucket in reverse.
- **The S3 â†’ Lambda trigger wiring crosses three files**: the notification lives on the bucket in `s3.tf` (`aws_s3_bucket_notification.uploads_trigger`), but it `depends_on` the invoke permission granted in `iam.tf` (`aws_lambda_permission.allow_s3`), and points at the Lambda resource defined in `lambda.tf`. Changing any one of these three in isolation will break the trigger.
- **The Pillow layer ARN is not hardcoded** â€” `lambda.tf` fetches it at plan time from the Klayers API (`data "http" "pillow_layer"`), filtered to `deployStatus == "latest"` for the region in `var.aws_region`. Changing `aws_region` in `variables.tf` automatically resolves a valid layer ARN for the new region; if Klayers doesn't publish a Pillow layer there, `terraform plan` will fail with an empty `one([...])` result.
- **Sizing config flows from `variables.tf` â†’ Lambda environment variables â†’ `handler.py`'s `SIZES` dict** (`THUMBNAIL_SIZE`, `MEDIUM_SIZE`, `LARGE_SIZE`). To change resize dimensions, edit the variables, not the Lambda code.
- **The resizer Lambda resizes all three variants concurrently** via a `ThreadPoolExecutor` in `handler.py` â€” this is intentional (I/O-bound S3 PUTs) and also narrows the window where the API Lambda's `/results` poll could see a partial set of variants. The `/results` handler explicitly waits for all three (`thumb`, `medium`, `large`) before reporting `"status": "done"`.
- **Both S3 buckets get random suffixes** (`random_id.suffix` in `main.tf`) for global uniqueness, and both fully block public access â€” nothing is served directly from S3; everything goes through presigned URLs generated by the API Lambda.
- **terraform.tfstate and .terraform/ are gitignored** (local backend only, no remote state) â€” do not expect state to be present after a fresh clone, and do not commit state files.
